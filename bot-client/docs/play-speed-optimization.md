# 机器人出牌速度优化

> 记录 bot-client 机器人「出牌偶尔卡死 / 每步 30~57 秒」问题的根因分析与分层决策管线设计。

## 1. 背景与问题

bot-client 是外部机器人,通过 WebSocket 接入 gateway 替代 C++ 内部机器人。接入真实
MiniMax 模型后,出现两类体验问题:

- **抢地主阶段**:轮到机器人叫地主时,界面要等 ~57 秒才有反应。
- **出牌阶段**:出牌很慢(30~57 秒/手),**偶尔整个回合卡死**不再前进。

根本约束:`MINIMAX_MODEL` 默认是 **MiniMax-M2.7,一个推理模型**,每次调用都会产出大量
思维链 token,壁钟延迟稳定在 30~57 秒。

## 2. 根因分析

### 2.1 结构化输出根本跑不通(先于速度的正确性 bug)

最初 `decision.ts` 用 Vercel AI SDK 的 `generateObject` 走结构化输出。把 MiniMax 从
Anthropic 兼容端点切到 OpenAI 兼容的 `/v1` 后,实测三种模式全部失败:

| 方式 | 结果 |
|------|------|
| `generateObject` tool 模式 | ❌ 失败(~57s 后报 could not parse the response) |
| `generateObject` json 模式 | ❌ 失败(~21s) |
| `generateText` + 自行解析 JSON | ✅ 成功,返回干净 JSON |

**结论**:MiniMax-M2.7 经 `/v1` 不支持 OpenAI 的 `response_format: json_schema` / tool
结构化输出。机器人每次决策都抛解析错误,被 agent 的 `try/catch` 吞掉 → 永远回退到
pass,游戏推不动。

修复:改用 `generateText` + `extractJsonObject()`(花括号配对解析,兼容 ` ```json ` 代码块
和推理模型的前置说明文字)+ zod 校验。详见 `src/decision.ts`。

### 2.2 抢地主不该用推理模型

抢地主只是 `grab / pass` 的二元判断,不值得用推理模型等几十秒。

### 2.3 出牌的延迟与卡死

- 延迟来自推理模型本身,无法通过提示词完全消除。
- 「卡死」来自:网络挂起或推理过久时,LLM 调用永不返回 →`actPlay` 永不完成 → 该机器人
  的回合冻结 → 全局停滞。

## 3. 设计:分层决策管线

核心洞察:`rules.ts` 的 `validSells()` 已经枚举出**所有合法出牌**,规则引擎完全能独立
产出合法且不差的牌。因此把决策拆成三层,**绝大多数回合根本不需要 LLM**,真正需要策略
的回合再调 LLM 且**加硬超时兜底**。

### 抢地主:瞬时启发式(不调 LLM)

`heuristicLandlord()`:炸弹 / 王炸 / 大王 → 直接抢;手握 ≥2 张高牌(2 或王)→ 抢;否则
不抢。瞬时返回。

### 出牌:`decidePlay()` 三层

| 层 | 触发条件 | 耗时 | 调 LLM |
|----|----------|------|--------|
| **第 0 层** | 0 个选项 → pass;领出唯一牌 → 出;跟牌唯一非炸弹 → 压;跟牌唯一炸弹 → 保留过牌 | 瞬时 | ❌ |
| **第 1 层** | 算出规则牌作为安全网(见下) | 瞬时 | ❌ |
| **第 2 层** | 多选策略回合,调 LLM,带硬超时 | ≤ 超时 | ✅ |

**第 1 层规则牌 `ruleBasedPlay()`**:
- 跟牌:优先用**最小的非炸弹**牌压制;若只能用炸弹/王炸 → 过牌保留炸弹。
- 领出:优先甩**最长的牌组**(顺子/连对)快速减手,否则出最小单牌;不主动拆炸弹。

**第 2 层 LLM**:
- 出牌专用模型默认 `MiniMax-M2.7-highspeed`(高速变体,`PLAY_MODEL` 可覆盖)。
- 用 `AbortSignal.timeout(PLAY_TIMEOUT_MS)` 加**硬超时**,真正取消请求。
- 超时 / 出错 / 解析失败 / 模型选了非法牌 → **立刻回退第 1 层规则牌**。

这套设计的关键收益:**强制回合秒出、永不挂死(最坏每回合超时封顶)、慢的尾部由规则牌
兜底**。

## 4. 优化效果展示

### 4.1 优化前 vs 优化后

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 抢地主 | LLM 推理 ~57s | **启发式,瞬时(<1ms)** |
| 强制/平凡出牌回合 | LLM 推理 30~57s | **规则短路,瞬时(<1ms)** |
| 多选策略回合 | M2.7,30~57s | highspeed,≤ `PLAY_TIMEOUT_MS`(默认 20s)封顶 |
| 卡死风险 | LLM 挂起则回合**永久冻结** | **永不卡死**:硬超时 → 规则牌兜底 |
| 决策正确性 | 解析失败被吞 → 永远 pass | `generateText` + 解析,失败有规则兜底 |

### 4.2 分层管线各档实测耗时

对 `decidePlay()` 各路径的真实计时(`real` 表示传入真实 provider 配置,第 0 层在调用
LLM 前短路,**不联网**):

```
耗时(ms)   | 场景                               | 决策
-----------|------------------------------------|------
    0.31 ms | 第0层 领出唯一牌(real)             | play 1张
    0.07 ms | 第0层 跟牌唯一非炸弹(real)         | play 1张
    0.05 ms | 第0层 跟牌唯一炸弹→过(real)        | pass
    0.27 ms | 第1层 多选规则牌(mock)             | play 5张
 1503.41 ms | 第2层 LLM超时1.5s→回退规则(real)   | play 1张
```

要点:第 0/1 层均为**亚毫秒**;第 2 层超时档精确停在 1503ms ≈ 设定的 1500ms 超时,
证明 `AbortSignal.timeout()` 真正取消了请求,而非空等。

### 4.3 真实对局决策日志(运行中机器人)

```
[robot_1] bidding decision: pass, 启发式:1 张王 → 不抢      ← 抢地主走启发式,瞬时
[robot_2] play decision: pass, 无合法出牌                    ← 第0层短路,瞬时
[robot_1] play decision: pass, 无合法出牌                    ← 第0层短路,瞬时
[robot_2] play decision: play, 规则:最小非炸弹压制          ← 规则牌兜底
```

`reason` 字段直接暴露了决策走的是哪条路径(启发式 / 无合法出牌 / 唯一合法牌 / 规则 /
LLM 自述理由),便于边玩边观察优化是否生效。

### 4.4 真实模型延迟参考

均为对真实 MiniMax `/v1` 端点的实测(壁钟):

| 场景 | 模型 | 延迟 | 备注 |
|------|------|------|------|
| 抢地主决策 | M2.7 | ~57s | 已改为启发式,现在瞬时 |
| 出牌(复杂局面,自己找牌型) | M2.7 | 30~57s | — |
| 出牌(受限提示,已列好选项,5 选项) | M2.7-highspeed | ~14~17s | — |
| 出牌(6 选项跟牌) | M2.7-highspeed | >25s | 被 25s 超时切断 → 回退规则 |

**诚实的现实**:highspeed 仍是推理模型,只是吞吐更快——简单局面 ~15s,复杂局面 25s+,
并非 3~8s。所以默认依赖**规则牌兜底处理慢的尾部**,而非指望模型每步都快。

## 5. 配置与调速旋钮

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `MINIMAX_MODEL` | 通用模型名 | `MiniMax-M2.7` |
| `PLAY_MODEL` | 出牌专用模型(高速变体) | `MiniMax-M2.7-highspeed` |
| `PLAY_TIMEOUT_MS` | 出牌 LLM 硬超时(毫秒),超时回退规则牌 | `20000` |

调速:

- **想更跟手** → 调小 `PLAY_TIMEOUT_MS`(如 `8000`):更多回合走规则,基本秒出,智能 ≈ 规则引擎。
- **想更聪明** → 调大(如 `30000`):LLM 参与更多,但偶尔慢。

改完重启机器人(`npm run dev`)生效。
